# Late-Start Window Fix — 2026-03-09T02:50:00Z

## Problem

When wsprrs started mid-WSPR-cycle, `even_minute_boundary(now)` returned the
start of the CURRENT (partially elapsed) window.  The first RTP packet had a
timestamp corresponding to the current wall-clock time, so the computed offset
was far into the window (e.g. 116 s × 12000 = 1,392,000 ≥ capacity_samples).
`ingest()` returned `BufferOverflow`, the window was dropped, the next packet
recreated it, and the cycle repeated indefinitely.

## Fix

Replaced `windows.entry(ssrc).or_insert_with(...)` with an explicit
`Entry::Occupied` / `Entry::Vacant` match.  On `Vacant`, the packet's RTP
timestamp is checked against the current even-minute boundary **before**
allocating the window.  If the packet falls outside `[0, capacity_samples)`,
the packet is silently discarded and the loop continues.

Once the next even-minute boundary rolls around, new packets will have offsets
starting near 0 and the window will be created normally.

## Log Output

With `RUST_LOG=debug`, skipped packets appear as:
```
DEBUG late start: skipping packet, waiting for next WSPR window
      ssrc=28125 approx_secs_into_window=117 capacity_secs=116
```
and then at the next even minute:
```
INFO  new audio window opened ssrc=28125 freq_hz=28124600 sample_rate=12000
```
