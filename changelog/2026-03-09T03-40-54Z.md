# 2026-03-09T03:40:54Z — Parallel wsprd decoding + wsprd 2.5+ parser

## Problems fixed

### 1. Serial decoding blocked audio capture

`decode_task` awaited each `wsprd` subprocess sequentially.  When ~20 bands
all sealed at the same 2-minute boundary, the decodes queued up and could
take minutes to drain — stalling the progress-bar display and delaying log
output for every band after the first.

### 2. wsprd 2.5+ output format rejected by parser

The installed `wsprd` binary emits the newer 2.5+ format on stdout:

```
<hash4>  <snr>  <dt>  <freq_MHz>  <drift>  <call>  [<grid4>]  <pwr>
```

e.g. `q0Bc  12  0.3  10.140114  0  W3POG FN20 23`

The old regex expected the 2.4 format:

```
<HHMM>  <snr>  <dt>  <freq_MHz>  <call>  <grid4>  <pwr>  [<drift>  <cycles>  <jitter>]
```

Key differences:
- Field[0]: decode-quality hash (not HHMM time)
- Field[4]: drift before callsign (not callsign before drift)
- Grid is absent for type-2 messages (e.g. `XDU/B4LHM 13`)
- `cycles`/`jitter` not emitted to stdout in 2.5+

## Changes

### `src/decode.rs` — new format-agnostic parser

- **Removed** `spot_regex()` / `OnceLock<Regex>` entirely.
- Replaced with a whitespace-split positional parser that **auto-detects**
  which format is in use:
  - `fields[4].parse::<i32>().is_ok()` → new format (drift at [4])
  - Otherwise → old format (callsign at [4])
- `parse_new_format`: derives `time_utc` from `window_start` (new format
  doesn't include the time); handles optional grid for type-2 messages;
  `decode_cycles`/`jitter` default to 0.
- `parse_old_format`: parses all 7–10 fields with optional trailing columns.
- Added `is_grid_square()` and `hhmm_utc()` helpers.
- Removed direct `regex` dependency from `Cargo.toml`.
- Updated tests: 6 new decode tests, 2 updated, total 43 passing.

### `src/main.rs` — per-window concurrent spawning

- Extracted `decode_window(window, cfg) -> anyhow::Result<Vec<WsprSpot>>`:
  handles WAV write (blocking pool) + wsprd subprocess for one window.
- Replaced the serial `decode_task` loop with a two-arm `select!`:
  - **`win_rx` arm**: `tokio::spawn(decode_window(...))` per window — returns
    immediately, all bands run in parallel.
  - **`spot_rx` arm**: collects results from an `mpsc::unbounded_channel` and
    serialises file I/O in a single place (no concurrent file writers).
- Audio capture in `buffer_task` is now completely unaffected by decode time.
- On a 16-core machine, all 20+ bands decode simultaneously rather than in
  a ~400-second serial queue.

## Impact

- All spots now logged correctly.
- Decode pipeline no longer competes with audio ingestion.
- All 43 unit tests pass; clippy clean.
