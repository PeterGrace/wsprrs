# Switch from IQ/.c2 to Mono Audio/WAV — 2026-03-09T02:44:51Z

## Root Cause

The original design assumed ka9q-radio would output raw **stereo IQ** (channels=2,
PT=123) for WSPR.  The standard and correct ka9q-radio WSPR config uses **USB
demodulation** with mono audio output (channels=1, PT=122, `mode=usb`).  No IQ
stream was ever present, so no audio windows were ever opened, and no progress
bars ever appeared.

## Architecture Change

| | Before | After |
|---|---|---|
| Expected input | Stereo IQ (`channels==2`) | Mono USB audio (`channels==1`) |
| Buffer type | `IqWindow` (interleaved I/Q) | `AudioWindow` (mono PCM) |
| Bytes per sample | 4 (2×i16 I+Q) | 2 (1×i16 mono) |
| Decode input | `.c2` binary (raw IQ float) | `.wav` (standard PCM) |
| wsprd invocation | `wsprd -c <file.c2>` | `wsprd -c -f <dial_MHz> <file.wav>` |
| Frequency handling | Embedded in .c2 header | `-f <dial_MHz>` flag to wsprd |

## Spot Frequency

For USB mode, `wsprd` computes spot frequency as:

```
spot_freq_MHz = dial_freq_MHz + audio_tone_Hz / 1_000_000
```

With `freq = 14m095600` in ka9q-radio config, a WSPR signal at 1500 Hz audio
is reported at 14.097100 MHz — correct for 20m WSPR.

## Files Changed

- **`src/buffer.rs`** — `IqWindow` → `AudioWindow`; mono capacity (no ×2);
  `ingest()` uses `chunks_exact(2)`; removed `to_f32()`, added `samples()`;
  stored `sample_rate_hz` field for WAV header
- **`src/status.rs`** — `is_iq_ready()` → `is_audio_ready()`: requires
  `channels==1`; updated tests
- **`src/decode.rs`** — replaced `write_c2_file()` + UTC helpers with
  `write_wav_file()`; updated `run_wsprd()` to pass `-f dial_freq_MHz`
- **`src/rtp.rs`** — renamed `iq_payload` → `payload` in `RtpPacket`
- **`src/main.rs`** — `IqWindow` → `AudioWindow` throughout; `is_audio_ready()`;
  updated `decode_task` to call `write_wav_file`
