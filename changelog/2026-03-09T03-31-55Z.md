# 2026-03-09T03:31:55Z — Fix: drop `-c` flag; filter `Writing ...` status lines

## Problem

Decoded WSPR spots were never logged because the `wsprd` output parser
rejected every output line with "skipping unparseable wsprd output line".

Two separate failures:

### 1. Wrong output format from `-c` flag

`wsprd` was invoked with `-c` (all candidates), which changes the stdout
format from the standard columnar report to a raw-candidate format:

```
p46h  35  0.2   3.570039  0  W3POG FN20 23
```

Fields: `<type_hash>  <cycles>  <dt>  <freq>  <drift>  <call>  <grid>  <pwr>`

This format:
- Has no `HHMM` time in column 1 (field 1 is alphanumeric, e.g. `p46h`)
- Has no calibrated SNR
- Fails the `^\s*(\d{4})` anchor in the spot regex

The standard format (without `-c`) is what the regex was written for:
```
HHMM  SNR  DT  Freq  Callsign  Grid  Pwr  Drift  Cycles  Jitter
```

### 2. `Writing ...` status lines not filtered

`wsprd` emits progress lines like `Writing 000000_0001.c2` to stdout.
These were reaching `parse_spot_line()` and returning `SpotParseFailed`.

## Fix (`src/decode.rs`)

1. **Removed the `-c` flag** from the `Command` builder in `run_wsprd()`.
   Without `-c`, wsprd outputs one clean calibrated spot line per decoded
   station in the expected 10-column format.

2. **Added `line.starts_with("Writing")` skip** in `parse_wsprd_output()`
   alongside the existing empty-line and `<DecodeFinished>` filters.

## Impact

- Spots are now parsed correctly and logged as NDJSON.
- With `-c` removed, each unique signal produces one line (best decode)
  rather than multiple candidate lines. This matches expected WSPR logging
  behaviour.
- All 37 unit tests pass; clippy clean.
